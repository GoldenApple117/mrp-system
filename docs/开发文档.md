## 二、认证与权限系统

### 2.1 JWT 认证流程

```
用户输入账号密码
  → POST /api/auth/login
  → bcrypt验证密码
  → 签发JWT (sub=username, role, approved, exp=24h)
  → 返回 { access_token, user }

前端:
  → localStorage存储token+user
  → axios请求拦截器自动附 Authorization: Bearer <token>
  → 响应拦截器: 401 → 清除token → 跳转/login
  → 路由守卫 beforeEach: 无token → /login
```

### 2.2 权限模型

| 角色 | is_approved | 权限 |
|------|:---:|------|
| admin | 1 | 全部功能 + 审批权限 |
| normal | 1 | 全部功能(已授权) |
| normal | 0 | 仅可见导航 → 内容区遮罩 → 需申请权限 |

### 2.3 权限申请流程

```
普通用户登录 → 看到遮罩 + "申请权限"按钮
  → POST /api/permissions/request
  → 管理员顶栏看到红色角标
  → 进入权限管理页 → 同意/拒绝
  → 同意 → 更新 user.is_approved=1
  → 用户刷新 → 遮罩消失
```

### 2.4 安全措施

- JWT Secret: 优先环境变量, 未设自动生成随机密钥
- 密码: bcrypt 4.0.1 哈希
- API保护: 所有业务路由 require_approved 依赖注入
- smtp_config.json: .gitignore 排除

## 五、MRP 核心引擎

### 5.1 算法流程

```
MPS(成品需求) → BOM展开(LLC分层) → 净需求计算
  → 批量规则应用(LFL/FOQ/EOQ/MULT)
  → 提前期倒推(下达日 = 需求日 - 提前期)
  → 库存分配(现有库存 → 安全库存)
  → 例外识别(缺料/超期)
  → 输出采购建议 + 生产建议
```

### 5.2 批量规则

| 规则 | 说明 |
|------|------|
| LFL | 按需定量，需求多少建议多少 |
| FOQ | 固定批量，不足一倍的向上取整 |
| EOQ | 经济批量，取整到批量数量 |
| MULT | 倍数批量，取批量数量的整数倍 |

### 6.1 数据来源

系统数据主要来自金山文档「三工位-15台测试」：

| 工作表 | 导入行数 |

| 视觉BOM | 15 |
| 量具工具类 | 46 |
| **合计** | **172+** |

### 6.2 导入架构

采用单BOM头 + 多模块行架构：
- 1个 BomHeader (product="三工位测试台（15台套）")
- 177行 BomLine，按 parent_item_id 区分为 6 个模块
- 系统通过 `parent_item_id → material_name` 来识别模块归属

### 7.1 APScheduler 定时任务

```python
# backend/app/services/scheduler.py
BackgroundScheduler + CronTrigger
默认: 每天 06:00 自动执行 MRP + 转PO
```
